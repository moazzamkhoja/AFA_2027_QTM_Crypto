## Key Discussion Points

### 1. Industry-aggregator research (initial pass)
Searched how Artemis Analytics, Token Terminal, and DeFiLlama define "economic activity."
All three use flow metrics (fees, revenue, volume/GMV) and keep TVL as a separate, parallel
stock metric — initial read: supports fees-over-TVL for Decision 1. (This read was later
refined — see #2.)

### 2. The fees-as-tax-revenue correction (the turning point)
The human rejected fees on a sharper theoretical ground: economic activity is the *value*
generated by a transaction; fees are the *cost* of facilitating it — structurally identical
to a government's tax revenue not being a measure of GDP. Re-deriving PQ from the paper's
own M·V = P·Q identity confirms this: P·Q is nominal GDP — the value of goods/services
exchanged, not the toll collected on the exchange. A protocol's fee rate is a discretionary,
governance-set parameter (fee-switch votes, fee-tier choices) — exactly like a country's tax
rate — so fees move for policy reasons unrelated to real usage. Re-reading DeFiLlama's own
language ("fees... show how much the protocol is *facilitating* in economic activity")
literally actually supports this correction rather than the earlier fees-positive read: a
toll booth facilitates billions in freight without its toll revenue measuring that freight's
value.

### 3. What TVL actually represents
Walked through TVL concretely by protocol type: AMM inventory (LP-supplied tradeable assets),
lending pool (loanable funds), liquid-staking AUM. In every case TVL is the **capital stock**
that enables activity — the K in a production function — not the output Y. This formalizes
the human's own "TVL organizes economic activity" intuition correctly: capital enables
production, it isn't the production. A high-TVL pool with no trading produces zero realized
economic activity that period.

### 4. The corrected PQ definition
PQ = **transacted value** — the dollar value of what actually moved through the contract:
swap/DEX volume for AMMs, loan-origination/borrow flow for lending, notional volume for
derivatives — sector-appropriate via the project's existing `sector` field (Entry 16), rather
than one universal proxy forced across heterogeneous protocol types. This is also, not
coincidentally, what the *original* NVT ratio (Willy Woo's Bitcoin metric, this paper's direct
namesake) used on the coin side: on-chain transaction *value*, never fees. TVL and fees are
both retained as secondary diagnostics (capital-stock control; cost-of-intermediation/take
rate), plus a new Volume/TVL turnover diagnostic — a protocol-level restatement of M·V=PQ
with TVL as M. Logged as Decisions Log Entry 30.

### 5. Etherscan feasibility question
The human asked whether Etherscan's existing Transfer-log mechanism (already proven in
Channel 1) could directly compute true transfer volume: pull all Transfer events for a
token, multiply by daily price. Mechanically yes — but Channel 1's escrow approach is fast
*because* it filters on one sparse counterparty address; pulling *all* transfers of an
actively-traded token removes that filter and will hit the `getLogs` 1,000-log cap far more
often, forcing much finer block-range bisection. Entry 24 already found full-supply Transfer
enumeration "orders of magnitude beyond" the free key for a different purpose (coin-age); this
session could not re-verify that with real numbers because the Cowork sandbox has no network
route to `api.etherscan.io` (confirmed directly — even a plain request to google.com fails the
same way, so it's a blanket sandbox egress restriction, not Etherscan-specific). There's also
an independent methodological concern: one swap can fire multiple Transfer events via routing,
and exchange-internal transfers add non-economic noise, so raw summation would need netting
even if fully extractable.

### 6. Decision
Rather than resolve scale by extrapolation, run a small bounded pilot in the next Claude Code
session (which has live Etherscan access): two tokens (UNI, AAVE), a 30-day window, measure
real call counts/timing, dollarize, and cross-check against DeFiLlama's reported DEX volume to
quantify the noise multiplier — see `04_code/CLAUDE_CODE_PHASE2_PQ_PILOT_PROMPT.md`. Decision
1 and Decision 2 in `PHASE2_PQ_DECISION_STATUS.md` stay open pending this pilot's findings.
